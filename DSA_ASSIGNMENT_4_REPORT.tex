\section{Task 1: Smart City Road Network Optimizer}

\subsection{Objective}
The objective of Task 1 is to process a city road graph in which each road stores both \texttt{cost} and \texttt{travelTime}. The implementation first validates the graph through \texttt{validateGraph()}, then constructs the minimum-cost spanning road network using \texttt{buildMST()}, finds the minimum-travel-time route using \texttt{shortestPath(S, T)}, compares the MST path against the optimal path using \texttt{compareNetworks(S, T)}, and finally identifies MST roads whose removal increases the MST cost using \texttt{criticalRoads()}.

\subsection{Approach}
The complete graph is managed by the class \texttt{CityGraph}. Each road is stored in the structure \texttt{Edge}, which contains \texttt{u}, \texttt{v}, \texttt{cost}, and \texttt{travelTime}. The full road list is stored in \texttt{edges}, while the roads accepted into the MST are stored in \texttt{mstEdges}. The adjacency structures \texttt{adjFull} and \texttt{adjMST} are used to represent the full graph and the MST graph using travel-time weights.

The function \texttt{validateGraph()} first checks whether the graph satisfies the chosen validation policy. After successful validation, \texttt{buildMST()} applies Kruskal's algorithm. The helper \texttt{sortEdgeIndicesByCost()} arranges edge indices by increasing \texttt{cost}, and the DSU helpers \texttt{initDSU()}, \texttt{findParent()}, and \texttt{unionSets()} are used to decide whether a road should be accepted or rejected. For shortest travel-time queries, \texttt{dijkstra()} is used on \texttt{adjFull}, and the final route is rebuilt by \texttt{reconstructPath()}. The function \texttt{compareNetworks(S, T)} evaluates the travel time on both \texttt{adjFull} and \texttt{adjMST}, while \texttt{criticalRoads()} removes one road from \texttt{mstEdges} at a time and rebuilds the MST cost to measure the impact.

\subsection{Graph Validation Policy}
The validation policy enforced by \texttt{validateGraph()} is:
\begin{itemize}
    \item self-loops are not allowed,
    \item duplicate roads between the same unordered pair are not allowed,
    \item both \texttt{cost} and \texttt{travelTime} must be non-negative,
    \item every node id must lie in the range \([0, N-1]\),
    \item the graph must be connected, checked through \texttt{countReachableFromZero()}.
\end{itemize}
If any rule fails, the function reports the first specific reason. Invalid values of \texttt{S} and \texttt{T} are handled inside \texttt{shortestPath(S, T)} and \texttt{compareNetworks(S, T)}.

\section{Task 2: Logistics Dispatch Centre}

\subsection{Objective}
The objective of Task 2 is to manage a live dispatch queue in which the next processed request must always have the highest \texttt{priority}, while still supporting update operations and shift-end sorting. The implementation validates the request list using \texttt{validateInput(requests)}, builds the live structure using \texttt{buildStructure(requests)}, removes the highest-priority request using \texttt{dispatchNext()}, changes an existing request by \texttt{updatePriority(request\_id, new\_priority)}, sorts records using \texttt{sortRecords(records, key)}, and returns the highest-priority requests through \texttt{topKRequests(k)} without changing the live heap.

\subsection{Approach}
The complete implementation is managed by the class \texttt{DispatchCentre}. Each request is stored in the structure \texttt{Request}, which contains \texttt{requestID}, \texttt{priority}, and \texttt{timestamp}. The live queue is stored in the vector \texttt{heap}, treated as an array-based max-heap. The helper \texttt{buildStructure(requests)} copies the input requests into \texttt{heap} and restores the heap property through bottom-up \texttt{siftDown()} calls.

Once the structure is built, \texttt{dispatchNext()} removes the root request and restores the heap property by moving the replacement downward. The function \texttt{updatePriority(request\_id, new\_priority)} first finds the request using \texttt{findIndexByID(request\_id)} and then calls either \texttt{siftUp()} or \texttt{siftDown()} depending on whether the priority increased or decreased. For shift-end record handling, \texttt{sortRecords(records, key)} calls \texttt{quicksortHelper(arr, lo, hi, key, partitionCount)} to sort the given vector in-place. The function \texttt{topKRequests(k)} does not modify \texttt{heap}; instead, it copies the heap into \texttt{tempHeap} and extracts the maximum \(k\) times from the copy.

\subsection{Data Handling Policy}
The policy enforced by \texttt{validateInput(requests)} and \texttt{updatePriority(request\_id, new\_priority)} is:
\begin{itemize}
    \item every \texttt{requestID} must be unique inside one input batch,
    \item every \texttt{priority} must lie in the range \([0, 100]\),
    \item an \texttt{updatePriority()} call is invalid if the id is unknown,
    \item an \texttt{updatePriority()} call is invalid if the new priority is outside \([0, 100]\).
\end{itemize}
The code prints the exact invalid reason and leaves the live structure unchanged when an invalid update is attempted.

\subsection{Time and Space Complexity Analysis}
Let \(n\) be the number of requests currently stored in the structure.

\begin{table}[H]
\centering
\scriptsize
\begin{tabularx}{\textwidth}{|p{3.55cm}|p{2.2cm}|p{2.0cm}|X|}
\hline
\textbf{Operation} & \textbf{Time Complexity} & \textbf{Space Complexity} & \textbf{Explanation} \\
\hline
\texttt{swapNodes()} & \(O(1)\) & \(O(1)\) & Swaps two requests using one temporary variable. \\
\hline
\texttt{siftUp()}, \texttt{siftDown()} & \(O(\log n)\) & \(O(1)\) & Only one heap path is traversed. \\
\hline
\texttt{findIndexByID()} & \(O(n)\) & \(O(1)\) & Request ids are searched through a linear scan on \texttt{heap}. \\
\hline
\texttt{validateInput()} & \(O(n^2)\) & \(O(1)\) & Duplicate ids are checked by comparing each request with all earlier requests. \\
\hline
\texttt{buildStructure()} & \(O(n)\) & \(O(n)\) & Bottom-up heapify is used after copying the requests into \texttt{heap}. \\
\hline
\texttt{dispatchNext()} & \(O(\log n)\) & \(O(1)\) & Root removal followed by one \texttt{siftDown()} call. \\
\hline
\texttt{updatePriority()} & \(O(n)\) & \(O(1)\) & The linear scan in \texttt{findIndexByID()} dominates the update. \\
\hline
\texttt{sortRecords()} / \texttt{quicksortHelper()} & average \(O(n\log n)\), worst \(O(n^2)\) & average \(O(\log n)\) & Randomized in-place quicksort uses only recursion stack space on average. \\
\hline
\texttt{topKRequests(k)} & \(O(n + k\log n)\) & \(O(n)\) & A copy of \texttt{heap} is made in \texttt{tempHeap}, then extract-max is applied \(k\) times. \\
\hline
\end{tabularx}
\end{table}
